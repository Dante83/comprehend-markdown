# comprehend-markdown

*تلمس المخطوطة وتتمتم بالتعويذة. وبعد دقائق أو ساعة (حسب ثقل المخطوطة)، ستدرك محتوى ملفات Markdown بأي لغة كانت.*

خادم MCP يقوم بترجمة ملف `README.md` الخاص بالمشروع إلى لغات أخرى، بالإضافة إلى مسار عمل (pipeline) مستقل يشغل حلقة من الكتابة والمراجعة باستخدام نموذج LM Studio محلي.

يمكن تهيئة كل من لغة المصدر واللغات المستهدفة — حيث أن الإنجليزية هي اللغة الافتراضية فقط. قم بتعيين `source_language` للترجمة *من* الصينية أو الإندونيسية أو أي لغة أخرى، و`target_languages` لاختيار اللغات التي سيتم التوزيع إليها (راجع [اختيار لغات المصدر والهدف](#اختيار-لغات-المصدر-والهدف)).

بالنسبة لأي مشروع مستهدف، يتوقع الخادم وجود الملفات التالية (ويقوم بإنشائها عند الحاجة):

```
<project-root>/docs/<source>/README.md  المصدر الأساسي (الإنجليزية افتراضياً)
<project-root>/docs/<lang>/README.md    النسخ المترجمة، نسخة لكل لغة
<project-root>/README.md                صفحة هبوط لاختيار اللغة (يتم إنشاؤها تلقائياً)
```

كما يعمل النظام مع المشاريع التي لم يتم نقل ملفاتها بعد — حيث يكون المصدر لا يزال في الجذر — إذ يُستخدم ملف `README.md` الموجود في الجذر كمصدر، وفي نهاية تشغيل الـ `pipeline` يتم نقله إلى `docs/<source>/` (مثلاً `docs/English/`) ويُستبدل بصفحة هبوط قصيرة ومولدة تلقائياً تربط جميع الترجمات المتاحة.

## الإعداد (Setup)

يتم التعامل مع كل شيء (إنشاء البيئة الافتراضية venv، تثبيت/مزامنة التبعيات) عبر ملف `run.sh` — لا توجد خطوة تثبيت منفصلة. يقرأ الملف الحزم من `requirements.txt` ويعيد تثبيتها في كل مرة يتم فيها التشغيل، لذا فإن جلب تبعيات جديدة يعني ببساطة تشغيله مرة أخرى.

إذا كنت ترغب في الإشارة إلى نموذج LM Studio محلي غير النموذج الافتراضي، قم بنسخ ملف الإعدادات النموذجي وتعديله:

```bash
cp config.local.json.example config.local.json
```

ملف `config.local.json` مُتجاهل بواسطة git ويقوم بتجاوز قيم `config.json` مفتاحاً بمفتاح، لذا يمكنك تعديل `lm_studio_url` / `model_name` / `api_key` محلياً دون المساس بالقيم الافتراضية المرفوعة في المستودع.

### اختيار لغات المصدر والهدف

يتحكم مفتاحان في الإعدادات باتجاه الترجمة، ويقوم كل من الخادم ومسار العمل (pipeline) بقراءتهما (من `config.py`) لضمان عدم التعارض:

- **`source_language`** — اللغة التي كُتب بها ملف `README.md` الأساسي، واسم المجلد الخاص به في `docs/<source_language>/`. القيمة الافتراضية هي `"English"`. يمكنك تعيينها إلى `"中文"` أو `"Bahasa Indonesia"` أو أي لغة أخرى للترجمة *من* تلك اللغة بدلاً من الإنجليزية.
- **`target_languages`** — قائمة اللغات التي يترجم إليها مسار العمل (pipeline). يتم تخطي أي إدخال يساوي `source_language` تلقائياً، لذا فإن ترك لغة المصدر في القائمة لا يسبب أي ضرر.

على سبيل المثال، لترجمة ملف README صيني إلى الإنجليزية والإسبانية:

```json
{
  "source_language": "中文",
  "target_languages": ["English", "Español"]
}
```

إذا تم حذف `target_languages` من الإعدادات، سيعود مسار العمل إلى القائمة المدمجة التي تحتوي على 18 لغة.

لا يهم مفتاح `api_key` إلا إذا قمت بتفعيل خيار "Require API Key" في إعدادات خادم المطور (Developer server) في LM Studio — خلاف ذلك، اتركه كـ `"lm-studio"`، حيث يتجاهله LM Studio. لاحظ أن هذا ينطبق فقط على وضع الـ `pipeline` وهو الجزء الوحيد هنا الذي يستدعي نقطة نهاية LM Studio المتوافقة مع OpenAI مباشرة؛ أما وضع الـ `serve` فلا يتحدث أبداً مع واجهة برمجة تطبيقات (API) الخاصة بـ LM Studio لأن LM Studio *هو* عميل MCP الذي يقوم باستدعائه.

## الاستخدام (Usage)

```bash
./run.sh serve    /absolute/path/to/project   # خادم MCP عبر stdio
./run.sh pipeline /absolute/path/to/project   # تشغيل main.py من البداية إلى النهاية
```

يتطلب كلا الوضعين مساراً **مطلقاً (absolute)** لمجلد المشروع الذي يحتوي على ملف `README.md` المراد ترجمته. في وضع الـ `pipeline` يمكنك حذف المسار وسيُطلب منك إدخاله، بشرط أنك تقوم بالتشغيل بشكل تفاعلي عبر الطرفية (terminal):

```bash
./run.sh pipeline
Enter absolute path to project folder: /absolute/path/to/project
```

(وضع الـ `serve` لا يطلب مدخلات أبداً — بمجرد بدئه، تصبح stdin/stdout هي قناة JSON-RPC الخاصة بـ MCP، كما أن مضيفي MCP يقومون بتشغيله بشكل غير تفاعلي على أي حال.)

### `serve` — كأداة لمضيف MCP

هذا هو المسار الذي يجب أن يشير إليه "الأمر" (`command`) في إعدادات خادم المضيف (مثل LM Studio)، مع وضع المسار المطلق للمشروع المستهدف كـ "وسيط ثابت" (fixed argument). يوفر الخادم ما يلي:

- **أداة (tool)** `write_readme(language, content)` — تكتب ملف `docs/<language>/README.md`.
- **أداة (tool)** `write_directory_readme(content)` — تكتب ملف `README.md` في الجذر (صفحة هبوط اختيار اللغة).
- **مورد (resource)** `docs://readme` — المصدر (`docs/<source_language>/README.md` أو ملف `README.md` في الجذر كبديل).
- **مورد (resource)** `docs://readme/{language}` — الترجمة الموجودة، إن وجدت.
- **مورد (resource)** `docs://dir_readme` — ملف `README.md` في الجذر.
- **مطالبات (prompts)** `translate_readme`, `critique_translation`, `rewrite_translation` — حلقة الترجمة الجديدة.
- **مطالبات (prompts)** `check_existing_readme`, `rewrite_from_existing_translation` — مسار التحديث: مقارنة ترجمة موجودة بالمصدر الحالي وترقيعها بأقل تغييرات ممكنة.
- **مطالبة (prompt)** `create_docs_language_directory` — بناء صفحة اختيار اللغة في الجذر من قائمة الترجمات المتاحة.

يقوم نموذج المضيف نفسه بتوجيه استدعاءات الأدوات؛ هذا الخادم يوفر فقط عمليات الإدخال والإخراج للملفات وقوالب المطالبات (prompt templates).

#### إضافته إلى LM Studio

توجد إعدادات MCP الخاصة بـ LM Studio في المسار `~/.lmstudio/mcp.json`. أضف إدخالاً مثل:

```json
{
  "mcpServers": {
    "comprehend-markdown": {
      "command": "/absolute/path/to/comprehend-markdown/run.sh",
      "args": [
        "serve",
        "/absolute/path/to/the/project/you/want/to/translate"
      ]
    }
  }
}
```

الوسيط الثاني في `args` ثابت عند وقت الاتصال — إعدادات LM Studio هي عبارة عن JSON ثابت لا يدعم المطالبات التفاعلية، لذا يجب أن يكون المسار الفعلي الذي تريد ترجمته، وليس مسار هذا المستودع إلا إذا كان هو المشروع المقصود بالترجمة.

### `pipeline` — مستقل، بدون مضيف MCP

يقوم بتشغيل حلقة (ترجمة $\rightarrow$ نقد $\rightarrow$ مراجعة) كاملة للغات المحددة في `target_languages` (القائمة الافتراضية تضم 18 لغة: الألمانية، الإسبانية، الفرنسية، الإيطالية، البولندية، البرتغالية، الروسية، الفيتنامية، التايلاندية، الصينية، اليابانية، الكورية، العربية، الهندية، البنغالية، الإندونيسية، الأردية، والنايجيرية — راجع [اختيار لغات المصدر والهدف](#اختيار-لغات-المصدر-والهدف) لتغيير القائمة أو لغة المصدر)، حيث يستدعي نقطة نهاية LM Studio المتوافقة مع OpenAI مباشرة لأدوار الكاتب والمراجع، ويقوم بتشغيل نسخة داخلية من `server.py` عبر stdio للقيام بعمليات الملفات. هذا الوضع مفيد للترجمة الجماعية دون الحاجة لاستخدام واجهة مستخدم LM Studio. يتناسب وقت التشغيل مع حجم المستند عند استخدام نموذج محلي — دقائق لملف README صغير، وقد تصل إلى ساعة تقريباً للملفات الكبيرة المقسمة.

اللغات التي تمتلك بالفعل ملف `docs/<language>/README.md` لا يتم إعادة ترجمتها من الصفر: يقوم المراجع بمقارنة الترجمة الموجودة بالمصدر الحالي، وفقط في حال حدوث تغيير، يقوم الكاتب بترقيعها بتعديلات طفيفة. يتم تخطي الترجمات المحدثة تماماً، لذا فإن إعادة تشغيل الـ pipeline بعد تعديل README يعيد معالجة الأجزاء القديمة فقط.

بعد الانتهاء من العمل على كل لغة، تنتهي العملية (إذا لزم الأمر) بنقل ملف `README.md` المصدر من الجذر إلى `docs/<source_language>/` وإعادة توليد ملف `README.md` في الجذر كصفحة اختيار لغة قصيرة تربط كل ترجمة غير فارغة.

يتطلب هذا الوضع تشغيل خادم LM Studio المحلي (راجع `config.json`) مع تحميل أي نموذج دردشة — حيث يقوم الـ pipeline بتوجيه النموذج عبر إكمالات نصية بسيطة ويقوم بكتابة الملفات بنفسه، لذا **لا** يحتاج النموذج إلى دعم استدعاءات الأدوات (tool calls) بنمط OpenAI (راجع "التعامل مع ملفات README الكبيرة" أدناه). ومع ذلك، يُنصح باختبار لغة واحدة قبل الاعتماد على تشغيل كامل للقائمة.

### التعامل مع ملفات README الكبيرة

تعتبر ترجمة ملف README كبير (مثل وثائق `A-Starry-Sky` / `a-restless-ocean` التي يتراوح حجمها بين 35-60 كيلوبايت) في عملية إكمال واحدة هي المخاطرة الرئيسية من حيث الموثوقية عند استخدام نموذج محلي: فقد تنفد رموز الإخراج (output tokens) أو سعة السياق في منتصف العملية، مما يؤدي إلى بتر النهاية بصمت. تم بناء وضع الـ `pipeline` لتجنب ذلك:

- **مصمم ليكون خالياً من الأدوات (Tool-free).** ينتج النموذج نصوصاً عادية فقط — كل ترجمة أو إعادة كتابة أو صفحة دليل تعود كـ "رد" نصي، ويقوم المنسق بحفظها بنفسه عبر أدوات `write_readme` / `write_directory_readme`. لا يُطلب من النموذج أبداً وضع مستند كامل داخل وسيط لاستدعاء أداة. في نماذج *الاستدلال* المحلية (مثل Gemma 4)، يؤدي هذا المسار إلى بتر JSON الخاص بالوسائط ويرفضه النظام بخطأ `peg-gemma4 format` / مخرجات مشوهة؛ لذا فإن إرجاع النص يتجاوز هذه المشكلة تماماً.
- **`max_tokens`** (الافتراضي `32768`) يتم إرساله صراحةً في كل عملية إكمال لضمان انتهاء القسم أو المسودة بالكامل بدلاً من قطعها عند القيمة الافتراضية الأصغر لـ LM Studio لكل طلب. يحذرك الـ pipeline إذا توقفت عملية الإكمال بسبب "الطول" (`length`) لتعرف أنه يجب عليك رفع هذه القيمة.
- **تقسيم الأقسام (Section chunking)** — عندما يتجاوز المصدر قيمة `chunk_threshold_chars` (الافتراضي `12000`) وتكون قيمة `chunk_translation` هي `true`، يتم تقسيم المصدر بناءً على عناوين Markdown الرئيسية (`## `) (مع مراعاة كتل الكود، بحيث لا يتم تقسيم الـ `##` الموجودة داخل كود)، ويتم ترجمة كل قسم في عملية إكمال مستقلة، ثم يقوم المنسق بتجميع القطع وحفظها. لا تضطر أي استدعاء واحد للنموذج إلى إصدار المستند كاملاً.
- **مراجعة لكل قسم** — عند تفعيل `review_sections` (الافتراضي `true`)، يخضع كل قسم لنفس حلقة (نقد $\rightarrow$ مراجعة) التي يستخدمها المسار الموحد، ولكن بنطاق يقتصر على ذلك القسم فقط: يقارن المراجع القسم المترجم بمصدره ويقوم الكاتب بالمراجعة حتى يوافق المراجع (أو يتم الوصول إلى `MAX_ITERATIONS`). وبما أن القسم صغير، تظل عملية المراجعة وإعادة الكتابة بعيدة تماماً عن سقف البتر الذي جعل مراجعة المستند الكامل لملف README كبير أمراً غير آمن.

يتعمد المسار المقسم **عدم** إجراء مراجعة ثانية للمستند كاملاً بعد ذلك — لأن إعادة إصدار ملف README كبير بالكامل في عملية إكمال واحدة سيعيد مشكلة البتر، وقد غطت المراجعة على مستوى الأقسام هذا الأمر بالفعل. قم بتعيين `review_sections` إلى `false` لترجمة المستندات الكبيرة بدون أي مراجعة، أو تعيين `chunk_translation` إلى `false` لفرض سلوك "المرة الواحدة" الأصلي (والذي لا يزال يشغل حلقة مراجعة كاملة للمستند). أما المسار الموحد للملفات التي تقل عن الحد المسموح به فيبقى دون تغيير.
