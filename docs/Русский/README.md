# comprehend-markdown

*Вы касаетесь свитка и произносите заклинание. Спустя несколько минут или час (в зависимости от веса свитка), вы начинаете понимать Markdown на любом языке.*

MCP-сервер, который переводит `README.md` проекта на другие языки, а также автономный пайплайн, реализующий цикл «автор/рецензент» с использованием локальной модели LM Studio.

Исходный и целевые языки настраиваются — английский выбран лишь по умолчанию. Установите `source_language`, чтобы переводить *с* китайского, индонезийского или любого другого языка, и `target_languages`, чтобы выбрать языки перевода (см. раздел [Выбор исходного и целевых языков](#выбор-исходного-и-целевых-языков)).

Для любого целевого проекта ожидается следующая структура (или она будет создана автоматически):

```
<project-root>/docs/<source>/README.md  канонический источник (по умолчанию английский)
<project-root>/docs/<lang>/README.md    переведенные версии, по одной на язык
<project-root>/README.md                стартовая страница с выбором языка (генерируется автоматически)
```

Проекты, которые еще не были мигрированы (исходный файл всё еще находится в корне), также поддерживаются: корневой `README.md` используется как источник, а по завершении работы `pipeline` он перемещается в `docs/<source>/` (например, `docs/English/`) и заменяется короткой сгенерированной страницей со ссылками на все доступные переводы.

## Установка

Все операции (создание venv, установка/синхронизация зависимостей) выполняются скриптом `run.sh` — отдельный шаг установки не требуется. Он считывает пакеты из `requirements.txt` и переустанавливает их при каждом запуске, поэтому для обновления зависимостей достаточно просто запустить скрипт снова.

Если вы хотите использовать локальную модель LM Studio, отличную от стандартной, скопируйте пример конфигурации и отредактируйте его:

```bash
cp config.local.json.example config.local.json
```

Файл `config.local.json` добавлен в `.gitignore` и переопределяет ключи `config.json`, поэтому вы можете локально настроить `lm_studio_url`, `model_name` или `api_key`, не затрагивая значения по умолчанию в репозитории.

### Выбор исходного и целевых языков

Направление перевода контролируют два ключа конфигурации, которые считываются и сервером, и пайплайном (из `config.py`), что исключает расхождения:

- **`source_language`** — язык, на котором написан ваш канонический `README.md`, а также название папки `docs/<source_language>/`. По умолчанию `"English"`. Установите `"中文"`, `"Bahasa Indonesia"` или любой другой язык, чтобы переводить *с* него.
- **`target_languages`** — список языков, на которые пайплайн выполняет перевод. Любая запись, совпадающая с `source_language`, автоматически пропускается, поэтому наличие исходного языка в списке безопасно.

Например, чтобы перевести китайский README на английский и испанский:

```json
{
  "source_language": "中文",
  "target_languages": ["English", "Español"]
}
```

Если `target_languages` не указан, пайплайн использует встроенный список из 18 языков.

Параметр `api_key` имеет значение только в том случае, если в настройках Developer server в LM Studio включена опция «Require API Key» — в противном случае оставьте его как `"lm-studio"`, что LM Studio игнорирует. Обратите внимание, что это относится только к режиму `pipeline`, так как только он напрямую обращается к OpenAI-совместимому эндпоинту LM Studio; режим `serve` не взаимодействует с API LM Studio напрямую, поскольку сама LM Studio выступает в роли MCP-клиента.

## Использование

```bash
./run.sh serve    /absolute/path/to/project   # stdio MCP сервер
./run.sh pipeline /absolute/path/to/project   # запуск main.py end-to-end
```

Оба режима требуют **абсолютного** пути к папке проекта, содержащей `README.md` для перевода. В режиме `pipeline` путь можно опустить — в этом случае он будет запрошен интерактивно (при запуске в терминале):

```bash
./run.sh pipeline
Enter absolute path to project folder: /absolute/path/to/project
```

(Режим `serve` никогда не запрашивает ввод — после запуска stdin/stdout становятся каналом JSON-RPC для MCP, а MCP-хосты в любом случае запускают его в неинтерактивном режиме.)

### `serve` — как инструмент MCP-хоста

Именно этот режим должен быть указан в поле `command` конфигурации MCP-сервера в хосте (например, в LM Studio), где абсолютный путь к целевому проекту передается как фиксированный аргумент. Сервер предоставляет:

- **tool** `write_readme(language, content)` — записывает файл `docs/<language>/README.md`
- **tool** `write_directory_readme(content)` — записывает корневой `README.md` (страницу выбора языка)
- **resource** `docs://readme` — исходный текст (`docs/<source_language>/README.md`, или корневой `README.md` в качестве запасного варианта)
- **resource** `docs://readme/{language}` — существующий перевод, если он есть
- **resource** `docs://dir_readme` — корневой `README.md`
- **prompts** `translate_readme`, `critique_translation`, `rewrite_translation` — цикл создания нового перевода
- **prompts** `check_existing_readme`, `rewrite_from_existing_translation` — путь обновления: сравнение существующего перевода с текущим источником и внесение минимальных правок
- **prompt** `create_docs_language_directory` — создание корневой страницы выбора языка на основе списка доступных переводов

Вызовы инструментов осуществляет модель самого хоста; данный сервер лишь обеспечивает ввод-вывод файлов и предоставляет шаблоны промптов.

#### Добавление в LM Studio

Конфигурация MCP в LM Studio находится по адресу `~/.lmstudio/mcp.json`. Добавьте запись следующего вида:

```json
{
  "mcpServers": {
    "comprehend-markdown": {
      "command": "/absolute/path/to/comprehend-markdown/run.sh",
      "args": [
        "serve",
        "/absolute/path/to/the/project/you/want/to/translate"
      ]
    }
  }
}
```

Второй аргумент в `args` фиксируется при подключении — конфигурация LM Studio представляет собой статический JSON без поддержки интерактивных запросов, поэтому там должен быть указан путь к проекту, который вы хотите перевести (а не путь к этому репозиторию, если только вы не переводите сам этот проект).

### `pipeline` — автономный режим без MCP-хоста

Самостоятельно запускает полный цикл «перевод $\rightarrow$ критика $\rightarrow$ правка» для настроенных `target_languages` (список по умолчанию из 18 языков: Deutsch, Español, Français, Italiano, Polski, Português, Русский, Tiếng Việt, ไทย, 中文, 日本語, 한국어, العربية, हिन्दी, বাংলা, Bahasa Indonesia, اردو, Naijá — см. [Выбор исходного и целевых языков](#выбор-исходного-и-целевых-языков), чтобы изменить список или исходный язык).

Пайплайн напрямую обращается к OpenAI-совместимому эндпоинту LM Studio для этапов написания и рецензирования, запуская внутреннюю копию `server.py` через stdio для работы с файлами. Это удобно для пакетного перевода без использования интерфейса LM Studio. Время выполнения зависит от размера документа и локальной модели — от нескольких минут для короткого README до часа и более для больших документов, разбитых на части.

Языки, для которых уже существует `docs/<language>/README.md`, не переводятся с нуля: рецензент сравнивает существующий перевод с текущим источником и только в случае изменений автор вносит минимальные правки. Актуальные переводы полностью пропускаются, поэтому повторный запуск пайплайна после редактирования README обновит только устаревшие части.

После завершения работы по каждому языку пайплайн (при необходимости) перемещает исходный `README.md` из корня в `docs/<source_language>/` и заново генерирует корневой `README.md` как страницу выбора языка со ссылками на все непустые переводы.

Требуется запущенный локальный сервер LM Studio (см. `config.json`) с любой загруженной чат-моделью. Пайплайн управляет моделью с помощью обычных текстовых дополнений (completions) и сам записывает файлы, поэтому модели **не** требуется поддержка вызовов инструментов в стиле OpenAI (см. раздел «Работа с большими README» ниже). Тем не менее, рекомендуется протестировать один язык перед запуском полного цикла для всего списка.

### Работа с большими README

Перевод большого README (например, документация `A-Starry-Sky` / `a-restless-ocean` объемом ~35–60 КБ) за один запрос — главный риск при использовании локальных моделей: они могут исчерпать лимит выходных токенов или контекста на полпути, из-за чего конец документа будет молча обрезан. Режим `pipeline` спроектирован так, чтобы избежать этой проблемы:

- **Отказ от инструментов по дизайну.** Модель всегда генерирует только простой текст — каждый перевод, правка и страница каталога возвращаются как обычный ответ, а оркестратор сам сохраняет их через инструменты сервера `write_readme` / `write_directory_readme`. Модели никогда не предлагается поместить весь документ в аргумент вызова инструмента. В локальных *reasoning*-моделях (например, Gemma 4) такой путь приводит к обрезанию JSON-аргумента и ошибке эндпоинта (`peg-gemma4 format` / malformed-output); возврат обычного текста полностью решает эту проблему.
- **`max_tokens`** (по умолчанию `32768`) явно передается при каждом запросе, чтобы раздел или черновик мог быть завершен полностью, не обрываясь на стандартном (более низком) лимите LM Studio. Пайплайн предупредит, если генерация всё равно остановилась из-за достижения лимита (`length`), чтобы вы могли увеличить этот параметр.
- **Разбиение на секции (chunking)** — если размер источника превышает `chunk_threshold_chars` (по умолчанию `12000`) и параметр `chunk_translation` установлен в `true`, источник разделяется по заголовкам Markdown верхнего уровня (`## `). При этом учитываются блоки кода (заголовки внутри них игнорируются). Каждая секция переводится отдельным запросом, после чего оркестратор собирает части воедино и сохраняет их. Таким образом, ни один вызов модели не требует генерации всего документа целиком.
- **Рецензирование по секциям** — при значении `review_sections` `true` (по умолчанию) каждая секция проходит через тот же цикл «критика $\rightarrow$ правка», что и в обычном режиме, но только в рамках данной секции: рецензент сравнивает переведенную часть с оригиналом, а автор правит её до тех пор, пока рецензент не будет доволен (или не будет достигнуто `MAX_ITERATIONS`). Поскольку секция мала, процесс проверки и правки остается далеко от порога обрезания текста.

При использовании разбиения на части второй полный обзор всего документа намеренно **не** проводится — повторная генерация всего большого README одним запросом снова привела бы к обрезанию текста, а посекционный проход уже решил эту задачу. Установите `review_sections` в `false`, чтобы переводить большие документы без рецензирования, или `chunk_translation` в `false`, чтобы принудительно использовать режим одного запроса (в котором всё еще работает полный цикл рецензирования всего документа). Для документов меньше порога поведение остается прежним.
