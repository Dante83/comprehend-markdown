# comprehend-markdown

*Коснитесь свитка, произнесите заклятие — и спустя мгновения (или час, если свиток окажется тяжелым), любой Markdown станет вам понятен на любом языке.*

MCP-сервер, который переводит `README.md` проекта на другие языки, а также автономный пайплайн, запускающий цикл «автор — рецензент» с использованием локальной модели в LM Studio.

Исходный и целевые языки настраиваются — английский выбран лишь по умолчанию. Установите `source_language`, чтобы переводить *с* китайского, индонезийского или любого другого языка, и укажите `target_languages`, чтобы выбрать языки перевода (см. раздел [Выбор исходного и целевых языков](#выбор-исходного-и-целевых-языков)).

Для любого целевого проекта ожидается следующая структура (или она будет создана автоматически):

```
<project-root>/docs/<source>/README.md  канонический источник (по умолчанию английский)
<project-root>/docs/<lang>/README.md    переведенные версии, по одной на язык
<project-root>/README.md                главная страница с выбором языка (генерируется автоматически)
```

Проекты, которые еще не были мигрированы (исходный файл всё еще находится в корне), также поддерживаются: корневой `README.md` используется как источник, а по завершении работы `pipeline` он перемещается в `docs/<source>/` (например, `docs/English/`) и заменяется короткой сгенерированной посадочной страницей со ссылками на все доступные переводы.

## Установка

Все операции (создание venv, установка и синхронизация зависимостей) выполняются через `run.sh` — отдельный шаг установки не требуется. Скрипт читает пакеты из `requirements.txt` и переустанавливает их при каждом запуске, поэтому для обновления зависимостей достаточно просто запустить его снова.

Если вы хотите использовать локальную модель LM Studio, отличную от стандартной, измените значения в `config.json`.

### Выбор исходного и целевых языков

За направление перевода в `config.json` отвечают два ключа конфигурации:

- **`source_language`** — язык, на котором написан ваш канонический `README.md`, а также название папки `docs/<source_language>/`. По умолчанию — `English`. Установите `中文`, `Indonesia` или любое другое значение, чтобы переводить *с* этого языка.
- **`target_languages`** — список языков, на которые пайплайн будет выполнять перевод. Любая запись, совпадающая с `source_language`, автоматически игнорируется, поэтому наличие исходного языка в списке безопасно.

Например, чтобы перевести китайский README на английский и испанский:

```json
{
  "source_language": "中文",
  "target_languages": ["English", "Español"]
}
```

Если `target_languages` не указан, пайплайн использует встроенный список из 18 языков.

Параметр `api_key` имеет значение только в том случае, если в настройках Developer server в LM Studio включена опция «Require API Key» — в противном случае оставьте значение `"lm-studio"`, которое LM Studio игнорирует. Обратите внимание, что это относится только к режиму `pipeline`, так как это единственная часть программы, которая напрямую обращается к OpenAI-совместимому эндпоинту LM Studio; режим `serve` не взаимодействует с API LM Studio напрямую, поскольку сама LM Studio выступает в роли MCP-клиента.

Здесь можно установить `max_tokens`, но я не уверен, учитывает ли это LM Studio на самом деле. Перед запуском скрипта обязательно установите значение не менее 24576 в настройках самой модели.

## Использование

```bash
./run.sh serve    /absolute/path/to/project   # stdio MCP-сервер
./run.sh pipeline /absolute/path/to/project   # запуск main.py end-to-end
```

Оба режима требуют указания **абсолютного** пути к папке проекта, содержащей `README.md` для перевода. В режиме `pipeline` путь можно опустить — в этом случае он будет запрошен интерактивно (при условии запуска в терминале):

```bash
./run.sh pipeline
Enter absolute path to project folder: /absolute/path/to/project
```

(Режим `serve` никогда не запрашивает ввод — после старта stdin/stdout становятся каналом MCP JSON-RPC, а MCP-хосты в любом случае запускают его в неинтерактивном режиме.)

### `serve` — как инструмент для MCP-хоста

Именно этот режим должен быть указан в поле `command` конфигурации MCP-сервера (например, в LM Studio), где абсолютный путь к целевому проекту передается как фиксированный аргумент. Сервер предоставляет:

- **tool** `write_readme(language, content)` — записывает файл `docs/<language>/README.md`
- **tool** `write_directory_readme(content)` — записывает корневой `README.md` (страницу выбора языка)
- **resource** `docs://readme` — исходный текст (`docs/<source_language>/README.md`, либо корневой `README.md` в качестве запасного варианта)
- **resource** `docs://readme/{language}` — существующий перевод, если он есть
- **resource** `docs://dir_readme` — корневой `README.md`
- **prompts** `translate_readme`, `critique_translation`, `rewrite_translation` — цикл создания нового перевода
- **prompts** `check_existing_readme`, `rewrite_from_existing_translation` — путь обновления: сравнение существующего перевода с текущим источником и внесение минимальных правок
- **prompt** `create_docs_language_directory` — создание главной страницы выбора языка на основе списка доступных переводов

Вызовы инструментов осуществляются моделью самого хоста; данный сервер лишь обеспечивает ввод-вывод файлов и предоставляет шаблоны промптов.

#### Добавление в LM Studio

Конфигурация MCP в LM Studio находится по адресу `~/.lmstudio/mcp.json`. Добавьте запись следующего вида:

```json
{
  "mcpServers": {
    "comprehend-markdown": {
      "command": "/absolute/path/to/comprehend-markdown/run.sh",
      "args": [
        "serve",
        "/absolute/path/to/the/project/you/want/to/translate"
      ]
    }
  }
}
```

Второй аргумент в `args` фиксируется при подключении — конфигурация LM Studio представляет собой статический JSON без поддержки интерактивных запросов, поэтому здесь должен быть указан путь к проекту, который вы хотите перевести (а не путь к этому репозиторию, если только вы не переводите сам этот проект).
