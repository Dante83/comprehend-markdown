# comprehend-markdown

*आप उस प्राचीन पत्रक (scroll) को स्पर्श करते हैं और मंत्र का उच्चारण करते हैं। कुछ मिनटों से लेकर एक घंटे बाद (दस्तावेज़ की लंबाई के आधार पर), आप किसी भी भाषा में मार्कडाउन (Markdown) को समझने लगते हैं।*

यह एक MCP सर्वर है जो किसी प्रोजेक्ट की `README.md` फ़ाइल का अन्य भाषाओं में अनुवाद करता है, साथ ही इसमें एक स्टैंडअलोन पाइपलाइन भी है जो स्थानीय LM Studio मॉडल का उपयोग करके लेखक/समीक्षक (writer/reviewer) लूप चलाती है।

स्रोत और लक्ष्य भाषाएँ दोनों कॉन्फ़िगर करने योग्य हैं — अंग्रेज़ी केवल डिफ़ॉल्ट है। किसी अन्य भाषा (जैसे चीनी, इंडोनेशियाई आदि) से अनुवाद करने के लिए `source_language` सेट करें, और यह किन भाषाओं में अनुवादित हो, इसके लिए `target_languages` चुनें (देखें [स्रोत और लक्ष्य भाषाओं का चयन करना](#स्रोत-और-लक्ष्य-भाषाओं-का-चयन-करना))।

किसी भी लक्षित प्रोजेक्ट के लिए, यह निम्नलिखित संरचना की अपेक्षा करता है (और आवश्यकतानुसार इसे बनाता है):

```
<project-root>/docs/<source>/README.md  मुख्य स्रोत (डिफ़ॉल्ट रूप से अंग्रेज़ी)
<project-root>/docs/<lang>/README.md    अनुवादित संस्करण, प्रति भाषा एक
<project-root>/README.md                भाषा-चयन लैंडिंग पेज (जनरेटेड)
```

ऐसे प्रोजेक्ट जो अभी तक माइग्रेट नहीं हुए हैं — जहाँ स्रोत अभी भी रूट (root) पर है — वे भी काम करते हैं: रूट `README.md` को स्रोत के रूप में उपयोग किया जाता है, और `pipeline` रन के अंत में इसे `docs/<source>/` (जैसे `docs/English/`) में ले जाया जाता है और इसकी जगह एक छोटा जनरेटेड लैंडिंग पेज बना दिया जाता है जो सभी उपलब्ध अनुवादों के लिंक देता है।

## सेटअप (Setup)

सब कुछ (`venv` निर्माण, डिपेंडेंसी इंस्टॉलेशन/सिंक) `run.sh` द्वारा संभाला जाता है — कोई अलग इंस्टॉलेशन स्टेप नहीं है। यह `requirements.txt` से पैकेज पढ़ता है और हर बार चलाने पर उन्हें फिर से इंस्टॉल करता है, इसलिए नई डिपेंडेंसीज़ जोड़ने का मतलब बस इसे दोबारा चलाना है।

यदि आप डिफ़ॉल्ट के बजाय किसी अन्य स्थानीय LM Studio मॉडल का उपयोग करना चाहते हैं, तो उदाहरण कॉन्फ़िग को कॉपी करें और उसे एडिट करें:

```bash
cp config.local.json.example config.local.json
```

`config.local.json` फ़ाइल gitignored है और यह `config.json` की कुंजियों (keys) को ओवरराइड करती है, इसलिए आप कमिट किए गए डिफ़ॉल्ट्स को छुए बिना स्थानीय रूप से `lm_studio_url` / `model_name` / `api_key` में बदलाव कर सकते हैं।

### स्रोत और लक्ष्य भाषाओं का चयन करना

दो कॉन्फ़िग कुंजियाँ अनुवाद की दिशा को नियंत्रित करती हैं, और सर्वर तथा पाइपलाइन दोनों उन्हें (`config.py` से) पढ़ते हैं, ताकि उनमें कोई विसंगति न हो:

- **`source_language`** — वह भाषा जिसमें आपकी मुख्य `README.md` लिखी गई है, और इसके `docs/<source_language>/` फ़ोल्डर का नाम। यह डिफ़ॉल्ट रूप से `"English"` है। किसी अन्य भाषा से अनुवाद करने के लिए इसे `"中文"`, `"Bahasa Indonesia"` या कुछ और सेट करें।
- **`target_languages`** — उन भाषाओं की सूची जिनमें पाइपलाइन अनुवाद करती है। `source_language` के बराबर कोई भी प्रविष्टि स्वचालित रूप से छोड़ दी जाती है, इसलिए सूची में स्रोत भाषा का होना हानिकारक नहीं है।

उदाहरण के लिए, चीनी README को अंग्रेज़ी और स्पेनिश में अनुवाद करने के लिए:

```json
{
  "source_language": "中文",
  "target_languages": ["English", "Español"]
}
```

यदि `target_languages` नहीं दिया गया है, तो पाइपलाइन अपनी अंतर्निहित 18-भाषाओं की सूची का उपयोग करती है।

`api_key` केवल तभी मायने रखती है जब आपने LM Studio की डेवलपर सर्वर सेटिंग्स में "Require API Key" चालू किया हो — अन्यथा इसे `"lm-studio"` रहने दें, जिसे LM Studio अनदेखा कर देता है। ध्यान दें कि यह केवल `pipeline` मोड पर लागू होता है, क्योंकि यही एकमात्र हिस्सा है जो सीधे LM Studio के OpenAI-संगत एंडपॉइंट को कॉल करता है; `serve` मोड कभी भी LM Studio के API से बात नहीं करता क्योंकि LM Studio स्वयं वह MCP क्लाइंट है जो इसे कॉल कर रहा है।

## उपयोग (Usage)

```bash
./run.sh serve    /absolute/path/to/project   # stdio MCP सर्वर
./run.sh pipeline /absolute/path/to/project   # main.py को एंड-टू-एंड चलाता है
```

दोनों मोड के लिए उस प्रोजेक्ट फ़ोल्डर का **पूर्ण (absolute)** पथ आवश्यक है जिसमें अनुवाद की जाने वाली `README.md` मौजूद है। `pipeline` मोड में, यदि आप टर्मिनल पर इंटरैक्टिव रूप से चला रहे हैं, तो आप इसे छोड़ सकते हैं और आपसे पथ पूछा जाएगा:

```bash
./run.sh pipeline
Enter absolute path to project folder: /absolute/path/to/project
```

(`serve` मोड कभी प्रॉम्प्ट नहीं करता — एक बार शुरू होने के बाद, stdin/stdout स्वयं MCP JSON-RPC चैनल बन जाते हैं, और MCP होस्ट इसे गैर-इंटरैक्टिव रूप से लॉन्च करते हैं।)

### `serve` — एक MCP होस्ट टूल के रूप में

एक MCP होस्ट (जैसे LM Studio) को अपने सर्वर `command` में इसी पथ को इंगित करना चाहिए, जिसमें लक्षित प्रोजेक्ट का पूर्ण पथ एक निश्चित तर्क (argument) के रूप में हो। यह निम्नलिखित उपलब्ध कराता है:

- **tool** `write_readme(language, content)` — (`docs/<language>/README.md` लिखता है)
- **tool** `write_directory_readme(content)` — (रूट `README.md` यानी भाषा-चयन लैंडिंग पेज लिखता है)
- **resource** `docs://readme` — (स्रोत फ़ाइल: `docs/<source_language>/README.md`, या रूट `README.md`)
- **resource** `docs://readme/{language}` — (मौजूदा अनुवाद, यदि कोई हो)
- **resource** `docs://dir_readme` — (रूट `README.md`)
- **prompts** `translate_readme`, `critique_translation`, `rewrite_translation` — (नया अनुवाद लूप)
- **prompts** `check_existing_readme`, `rewrite_from_existing_translation` — (अपडेट पथ: मौजूदा अनुवाद की तुलना वर्तमान स्रोत से करना और न्यूनतम परिवर्तनों के साथ उसे पैच करना)
- **prompt** `create_docs_language_directory` — (उपलब्ध अनुवादों की सूची से रूट भाषा-चयन पेज बनाना)

होस्ट का अपना मॉडल टूल कॉल्स को संचालित करता है; यह सर्वर केवल फ़ाइल I/O और प्रॉम्प्ट टेम्पलेट्स प्रदान करता है।

#### इसे LM Studio में जोड़ना

LM Studio का MCP कॉन्फ़िग `~/.lmstudio/mcp.json` पर होता है। इसमें इस तरह की प्रविष्टि जोड़ें:

```json
{
  "mcpServers": {
    "comprehend-markdown": {
      "command": "/absolute/path/to/comprehend-markdown/run.sh",
      "args": [
        "serve",
        "/absolute/path/to/the/project/you/want/to/translate"
      ]
    }
  }
}
```

दूसरा `args` प्रविष्टि कनेक्शन के समय निश्चित होता है — LM Studio का कॉन्फ़िग एक स्थिर JSON है जिसमें इंटरैक्टिव प्रॉम्प्ट सपोर्ट नहीं है, इसलिए यह वही वास्तविक पथ होना चाहिए जिसे आप अनुवादित करना चाहते हैं।

### `pipeline` — स्टैंडअलोन, बिना MCP होस्ट के

यह कॉन्फ़िगर की गई `target_languages` (डिफ़ॉल्ट 18 भाषाओं की सूची: Deutsch, Español, Français, Italiano, Polski, Português, Русский, Tiếng Việt, ไทย, 中文, 日本語, 한국어, العربية, हिन्दी, বাংলা, Bahasa Indonesia, اردو, Naijá — सूची या स्रोत भाषा बदलने के लिए [स्रोत और लक्ष्य भाषाओं का चयन करना](#स्रोत-और-लक्ष्य-भाषाओं-का-चयन-करना) देखें) के लिए पूर्ण अनुवाद $\rightarrow$ समीक्षा $\rightarrow$ संशोधन लूप चलाता है। यह लेखक/समीक्षक चरणों के लिए सीधे LM Studio के OpenAI-संगत एंडपॉइंट को कॉल करता है और फ़ाइल I/O करने के लिए stdio पर `server.py` की अपनी आंतरिक कॉपी चलाता है। यह बिना LM Studio UI का उपयोग किए बैच-अनुवाद के लिए उपयोगी है। रनटाइम स्थानीय मॉडल पर दस्तावेज़ के आकार के साथ बढ़ता है — छोटे README के लिए कुछ मिनट, और बड़े चंक्ड (chunked) दस्तावेज़ों के लिए लगभग एक घंटा।

जिन भाषाओं के लिए पहले से `docs/<language>/README.md` मौजूद है, उन्हें शुरू से अनुवादित नहीं किया जाता: समीक्षक मौजूदा अनुवाद की तुलना वर्तमान स्रोत से करता है और केवल तभी जब कुछ बदला हो, लेखक उसे न्यूनतम संपादन के साथ पैच करता है। अपडेटेड अनुवादों को पूरी तरह छोड़ दिया जाता है, इसलिए README एडिट करने के बाद पाइपलाइन को फिर से चलाने पर केवल पुराने हिस्से ही दोबारा प्रोसेस होते हैं।

प्रति-भाषा कार्य के बाद, रन समाप्त होता है (यदि आवश्यक हो) रूट-लेवल स्रोत `README.md` को `docs/<source_language>/` में ले जाकर और रूट `README.md` को एक छोटे भाषा-चयन पेज के रूप में पुन: उत्पन्न करके जो हर गैर-खाली अनुवाद से लिंक करता है।

इसके लिए LM Studio का स्थानीय सर्वर (देखें `config.json`) किसी भी चैट मॉडल के साथ लोड होकर चलना चाहिए — पाइपलाइन मॉडल को प्लेन टेक्स्ट कम्पलीशन्स के साथ संचालित करती है और फ़ाइल राइट्स स्वयं करती है, इसलिए मॉडल को OpenAI-स्टाइल टूल कॉल्स का समर्थन करने की आवश्यकता **नहीं** है (नीचे "Large-README handling" देखें)। फिर भी पूरी सूची पर भरोसा करने से पहले एक भाषा का परीक्षण करना उचित है।

### Large-README हैंडलिंग

एक बड़े README (जैसे `A-Starry-Sky` / `a-restless-ocean` दस्तावेज़ जो ~35–60 KB के हैं) को एक ही कम्पलीशन में अनुवादित करना स्थानीय मॉडल पर मुख्य विश्वसनीयता जोखिम है: यह बीच में आउटपुट टोकन या कॉन्टेक्स्ट से बाहर हो जाता है और अंत का हिस्सा चुपचाप कट (truncate) जाता है। `pipeline` मोड इसी से बचने के लिए बनाया गया है:

- **डिज़ाइन द्वारा टूल-मुक्त (Tool-free by design):** मॉडल केवल प्लेन टेक्स्ट उत्पन्न करता है — हर अनुवाद, संशोधन और डायरेक्टरी पेज उसके उत्तर के रूप में आता है, और ऑर्केस्ट्रेटर इसे सर्वर के `write_readme` / `write_directory_readme` टूल्स के माध्यम से स्वयं सहेजता है। यह मॉडल से पूरे दस्तावेज़ को टूल-कॉल तर्क (argument) में डालने के लिए नहीं कहता। स्थानीय *रीज़निंग* मॉडल्स (जैसे Gemma 4) पर, वह तरीका तर्क JSON को काट देता है और एंडपॉइंट इसे `peg-gemma4 format` / malformed-output त्रुटि के साथ अस्वीकार कर देता है; टेक्स्ट वापस करना इस समस्या को पूरी तरह से टाल देता है।
- **`max_tokens`** (डिफ़ॉल्ट `32768`) हर कम्पलीशन पर स्पष्ट रूप से भेजा जाता है ताकि एक पूरा सेक्शन/ड्राफ्ट समाप्त हो सके, बजाय इसके कि वह LM Studio के छोटे प्रति-अनुरोध डिफ़ॉल्ट पर कट जाए। यदि कोई कम्पलीशन अभी भी `length` पर रुकता है, तो पाइपलाइन चेतावनी देती है ताकि आप इसे बढ़ा सकें।
- **सेक्शन चंकिंग (Section chunking)** — जब स्रोत `chunk_threshold_chars` (डिफ़ॉल्ट `12000`) से अधिक हो जाता है और `chunk_translation` `true` होता है, तो स्रोत को टॉप-लेवल (`## `) मार्कडाउन हेडिंग्स पर विभाजित किया जाता है (यह फेंस-अवेयर है, इसलिए कोड ब्लॉक के अंदर `##` को नहीं छुआ जाता है), प्रत्येक सेक्शन का अनुवाद अपने स्वयं के कम्पलीशन में किया जाता है, और ऑर्केस्ट्रेटर टुकड़ों को जोड़कर सहेजता है। किसी भी एकल मॉडल कॉल को पूरा दस्तावेज़ उत्सर्जित करने की आवश्यकता नहीं होती।
- **प्रति-सेक्शन समीक्षा (Per-section review)** — `review_sections` `true` (डिफ़ॉल्ट) होने पर, प्रत्येक सेक्शन उसी समीक्षा $\rightarrow$ संशोधन लूप से गुजरता है जिसका उपयोग सिंगल-शॉट पाथ करता है, बस वह उस एक सेक्शन तक सीमित होता है: समीक्षक अनुवादित सेक्शन की तुलना उसके स्रोत से करता है और लेखक तब तक संशोधित करता है जब तक कि समीक्षक सहमत न हो जाए (या `MAX_ITERATIONS` पूरा न हो जाए)। चूंकि एक सेक्शन छोटा होता है, इसलिए समीक्षा और पुनर्लेखन उस ट्रंकेशन सीमा से बहुत दूर रहते हैं जिसने बड़े README की पूर्ण-दस्तावेज़ समीक्षा को असुरक्षित बना दिया था।

चंक्ड पाथ जानबूझकर बाद में दूसरी बार पूरे दस्तावेज़ की समीक्षा **नहीं** करता है — एक ही कम्पलीशन में पूरे बड़े README को फिर से उत्सर्जित करना उसी ट्रंकेशन को वापस ले आएगा, और प्रति-सेक्शन पास ने पहले ही इसे कवर कर लिया है। बिना किसी समीक्षा के बड़े दस्तावेज़ों का अनुवाद करने के लिए `review_sections` को `false` पर सेट करें, या मूल सिंगल-शॉट व्यवहार (जो अभी भी पूर्ण-दस्तावेज़ समीक्षा लूप चलाता है) को मजबूर करने के लिए `chunk_translation` को `false` पर सेट करें। थ्रेशोल्ड से नीचे का सिंगल-शॉट पाथ अपरिवर्तित रहता है।
