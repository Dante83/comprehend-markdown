# comprehend-markdown

*आप उस प्राचीन पत्र (scroll) को छूते हैं और मंत्र का उच्चारण करते हैं। कुछ मिनटों से लेकर एक घंटे बाद (दस्तावेज़ की लंबाई के अनुसार), आप किसी भी भाषा में Markdown को समझने लगते हैं।*

यह एक MCP सर्वर है जो किसी प्रोजेक्ट के `README.md` का अन्य भाषाओं में अनुवाद करता है, साथ ही इसमें एक स्टैंडअलोन पाइपलाइन भी है जो स्थानीय LM Studio मॉडल का उपयोग करके राइटर/रिव्यूअर लूप (writer/reviewer loop) चलाती है।

स्रोत (source) और लक्ष्य (target) भाषाएँ दोनों कॉन्फ़िगर की जा सकती हैं — अंग्रेजी केवल डिफॉल्ट है। चीनी, इंडोनेशियाई या किसी अन्य भाषा से अनुवाद करने के लिए `source_language` सेट करें, और यह किन भाषाओं में अनुवादित हो, इसके लिए `target_languages` चुनें (देखें [स्रोत और लक्ष्य भाषाओं का चयन करना](#choosing-source-and-target-languages))।

किसी भी टारगेट प्रोजेक्ट के लिए, यह निम्नलिखित संरचना की अपेक्षा करता है (और आवश्यकतानुसार इसे बनाता है):

```
<project-root>/docs/<source>/README.md  आधिकारिक स्रोत (डिफॉल्ट रूप से अंग्रेजी)
<project-root>/docs/<lang>/README.md    अनुवादित संस्करण, प्रति भाषा एक
<project-root>/README.md                लैंग्वेज-पिकर लैंडिंग पेज (जनरेटेड)
```

ऐसे प्रोजेक्ट जो अभी तक माइग्रेट नहीं हुए हैं — जहाँ स्रोत अभी भी रूट (root) पर है — वे भी काम करते हैं: रूट `README.md` को स्रोत के रूप में उपयोग किया जाता है, और `pipeline` रन के अंत में इसे `docs/<source>/` (जैसे `docs/English/`) में ले जाया जाता है और एक छोटे जनरेटेड लैंडिंग पेज द्वारा बदल दिया जाता है जो सभी उपलब्ध अनुवादों के लिंक प्रदान करता है।

## सेटअप (Setup)

सब कुछ (venv निर्माण, डिपेंडेंसी इंस्टॉल/सिंक) `run.sh` द्वारा संभाला जाता है — कोई अलग इंस्टॉलेशन स्टेप नहीं है। यह `requirements.txt` से पैकेज पढ़ता है और हर बार चलाने पर उन्हें री-इंस्टॉल करता है, इसलिए नई डिपेंडेंसीज़ लाने का मतलब बस इसे दोबारा चलाना है।

यदि आप डिफॉल्ट के अलावा किसी अन्य स्थानीय LM Studio मॉडल का उपयोग करना चाहते हैं, तो `config.json` में मानों (values) को बदलें।

### स्रोत और लक्ष्य भाषाओं का चयन करना (Choosing source and target languages)

`config.json` में दो कॉन्फ़िगरेशन कीज़ अनुवाद की दिशा को नियंत्रित करती हैं:

- **`source_language`** — वह भाषा जिसमें आपका आधिकारिक `README.md` लिखा गया है, और इसके `docs/<source_language>/` फ़ोल्डर का नाम। यह डिफॉल्ट रूप से `English` है। इसे `中文`, `Indonesia`, या किसी अन्य भाषा पर सेट करें ताकि उस भाषा से अनुवाद किया जा सके।
- **`target_languages`** — उन भाषाओं की सूची जिनमें पाइपलाइन अनुवाद करती है। `source_language` के बराबर कोई भी प्रविष्टि स्वचालित रूप से छोड़ दी जाती है, इसलिए सूची में स्रोत भाषा को रखना सुरक्षित है।

उदाहरण के लिए, चीनी README का अंग्रेजी और स्पेनिश में अनुवाद करने के लिए:

```json
{
  "source_language": "中文",
  "target_languages": ["English", "Español"]
}
```

यदि `target_languages` नहीं दिया गया है, तो पाइपलाइन अपनी अंतर्निहित 18-भाषाओं की सूची का उपयोग करती है।

`api_key` केवल तभी मायने रखती है जब आपने LM Studio की डेवलपर सर्वर सेटिंग्स के तहत "Require API Key" को चालू किया हो — अन्यथा इसे `"lm-studio"` रहने दें, जिसे LM Studio अनदेखा कर देता है। ध्यान दें कि यह केवल `pipeline` मोड पर लागू होता है, क्योंकि यही एकमात्र हिस्सा है जो सीधे LM Studio के OpenAI-संगत एंडपॉइंट को कॉल करता है; `serve` मोड कभी भी LM Studio के API से बात नहीं करता क्योंकि LM Studio स्वयं वह MCP क्लाइंट है जो इसे कॉल कर रहा है।

यहाँ `max_tokens` सेट करने का विकल्प उपलब्ध है, लेकिन मैं अनिश्चित हूँ कि LM Studio वास्तव में इसका सम्मान करता है या नहीं। इस स्क्रिप्ट को चलाने से पहले मॉडल में ही इसे कम से कम 24576 के मान पर सेट करना सुनिश्चित करें।

## उपयोग (Usage)

```bash
./run.sh serve    /absolute/path/to/project   # stdio MCP सर्वर
./run.sh pipeline /absolute/path/to/project   # main.py को एंड-टू-एंड चलाता है
```

दोनों मोड के लिए उस प्रोजेक्ट फ़ोल्डर का **एब्सोल्यूट (पूर्ण)** पाथ आवश्यक है जिसमें अनुवाद किया जाने वाला `README.md` मौजूद है। `pipeline` मोड में, यदि आप टर्मिनल पर इंटरैक्टिव रूप से चला रहे हैं, तो आप इसे छोड़ सकते हैं और इसके बजाय प्रॉम्प्ट के माध्यम से पाथ दे सकते हैं:

```bash
./run.sh pipeline
Enter absolute path to project folder: /absolute/path/to/project
```

(`serve` मोड कभी प्रॉम्प्ट नहीं करता — एक बार शुरू होने के बाद, stdin/stdout स्वयं MCP JSON-RPC चैनल बन जाते हैं, और MCP होस्ट इसे गैर-इंटरैक्टिव रूप से लॉन्च करते हैं।)

### `serve` — एक MCP होस्ट टूल के रूप में

एक MCP होस्ट (जैसे LM Studio) को अपने सर्वर `command` में इसी पाथ को इंगित करना चाहिए, जिसमें टारगेट प्रोजेक्ट का एब्सोल्यूट पाथ एक फिक्स्ड आर्गुमेंट के रूप में हो। यह निम्नलिखित उपलब्ध कराता है:

- **tool** `write_readme(language, content)` — `docs/<language>/README.md` लिखता है
- **tool** `write_directory_readme(content)` — रूट `README.md` (लैंग्वेज-पिकर लैंडिंग पेज) लिखता है
- **resource** `docs://readme` — स्रोत दस्तावेज़ (`docs/<source_language>/README.md`, या रूट `README.md`)
- **resource** `docs://readme/{language}` — मौजूदा अनुवाद, यदि कोई हो
- **resource** `docs://dir_readme` — रूट `README.md`
- **prompts** `translate_readme`, `critique_translation`, `rewrite_translation` — नए अनुवाद का लूप
- **prompts** `check_existing_readme`, `rewrite_from_existing_translation` — अपडेट पाथ: मौजूदा अनुवाद की तुलना वर्तमान स्रोत से करना और न्यूनतम परिवर्तनों के साथ उसे पैच करना
- **prompt** `create_docs_language_directory` — उपलब्ध अनुवादों की सूची से रूट लैंग्वेज-पिकर पेज बनाना

होस्ट का अपना मॉडल टूल कॉल्स को संचालित करता है; यह सर्वर केवल फ़ाइल I/O और प्रॉम्प्ट टेम्पलेट्स प्रदान करता है।

#### इसे LM Studio में जोड़ना

LM Studio का MCP कॉन्फ़िगरेशन `~/.lmstudio/mcp.json` पर होता है। इसमें इस तरह की प्रविष्टि जोड़ें:

```json
{
  "mcpServers": {
    "comprehend-markdown": {
      "command": "/absolute/path/to/comprehend-markdown/run.sh",
      "args": [
        "serve",
        "/absolute/path/to/the/project/you/want/to/translate"
      ]
    }
  }
}
```

दूसरा `args` प्रविष्टि कनेक्शन के समय तय हो जाता है — LM Studio का कॉन्फ़िगरेशन एक स्टैटिक JSON है जिसमें इंटरैक्टिव प्रॉम्प्ट सपोर्ट नहीं है, इसलिए यह वही वास्तविक पाथ होना चाहिए जिसका आप अनुवाद करना चाहते हैं, न कि इस रेपो का अपना पाथ (जब तक कि आप इसी प्रोजेक्ट का अनुवाद न करना चाहें)।
