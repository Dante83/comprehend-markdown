# comprehend-markdown

*เมื่อคุณสัมผัสคัมภีร์และร่ายมนตรา เพียงไม่กี่นาทีหรือหนึ่งชั่วโมงต่อมา (ขึ้นอยู่กับความหนาของคัมภีร์) คุณจะสามารถเข้าใจเนื้อหา Markdown ได้ในทุกภาษา*

MCP server สำหรับแปลไฟล์ `README.md` ของโปรเจกต์ให้เป็นภาษาต่างๆ พร้อมด้วย pipeline แบบ standalone ที่รันลูปการเขียนและตรวจทาน (writer/reviewer loop) โดยใช้โมเดลจาก LM Studio ในเครื่องของคุณ

คุณสามารถกำหนดค่าทั้งภาษาต้นทางและภาษาปลายทางได้ โดยมีภาษาอังกฤษเป็นค่าเริ่มต้น คุณสามารถตั้งค่า `source_language` เพื่อแปล *ออกจาก* ภาษาจีน, อินโดนีเซีย หรือภาษาอื่นๆ และตั้งค่า `target_languages` เพื่อเลือกภาษาที่ต้องการให้แปลออกไป (ดูรายละเอียดที่ [การเลือกภาษาต้นทางและปลายทาง](#การเลือกภาษาต้นทางและปลายทาง))

สำหรับโปรเจกต์เป้าหมาย ระบบจะคาดหวัง (และสร้างขึ้นหากจำเป็น) ดังนี้:

```
<project-root>/docs/<source>/README.md  แหล่งข้อมูลหลัก/ต้นฉบับ (ค่าเริ่มต้นคือ English)
<project-root>/docs/<lang>/README.md    เวอร์ชันที่แปลแล้ว แยกตามภาษา
<project-root>/README.md                หน้าแลนดิ้งเพจสำหรับเลือกภาษา (สร้างโดยอัตโนมัติ)
```

หากเป็นโปรเจกต์ที่ยังไม่ได้ย้ายโครงสร้าง (ไฟล์ต้นฉบับยังอยู่ที่ root) ก็สามารถใช้งานได้เช่นกัน โดยระบบจะใช้ `README.md` ที่ root เป็นต้นฉบับ และเมื่อรัน `pipeline` จนเสร็จสิ้น ไฟล์ดังกล่าวจะถูกย้ายไปที่ `docs/<source>/` (เช่น `docs/English/`) และถูกแทนที่ด้วยหน้าแลนดิ้งเพจสั้นๆ ที่รวมลิงก์ไปยังคำแปลทุกภาษาที่มีอยู่

## การติดตั้ง (Setup)

ทุกขั้นตอน (การสร้าง venv, การติดตั้ง/ซิงค์ dependency) ถูกจัดการโดย `run.sh` จึงไม่มีขั้นตอนการติดตั้งแยกต่างหาก ระบบจะอ่านแพ็กเกจจาก `requirements.txt` และติดตั้งใหม่ทุกครั้งที่รัน ดังนั้นหากต้องการอัปเดต dependency เพียงแค่รันสคริปต์นี้อีกครั้ง

หากคุณต้องการระบุโมเดลใน LM Studio ตัวอื่นนอกเหนือจากค่าเริ่มต้น ให้คัดลอกไฟล์ตัวอย่างการตั้งค่าและแก้ไขดังนี้:

```bash
cp config.local.json.example config.local.json
```

ไฟล์ `config.local.json` จะถูก gitignore และจะเขียนทับค่าใน `config.json` แบบทีละคีย์ ดังนั้นคุณจึงสามารถปรับแต่ง `lm_studio_url` / `model_name` / `api_key` ในเครื่องของคุณได้โดยไม่ต้องแก้ไขค่าเริ่มต้นที่ commit ไว้

### การเลือกภาษาต้นทางและปลายทาง

มีคีย์การตั้งค่าสองตัวที่ควบคุมทิศทางการแปล ซึ่งทั้ง server และ pipeline จะอ่านค่าเหล่านี้ (จาก `config.py`) เพื่อให้ทำงานสอดคล้องกัน:

- **`source_language`** — ภาษาที่ใช้เขียน `README.md` ต้นฉบับ และเป็นชื่อโฟลเดอร์ `docs/<source_language>/` ค่าเริ่มต้นคือ `"English"` คุณสามารถเปลี่ยนเป็น `"中文"`, `"Bahasa Indonesia"` หรือภาษาอื่นๆ เพื่อแปล *ออกจาก* ภาษานั้นๆ
- **`target_languages`** — รายชื่อภาษาที่ pipeline จะแปล *ไปเป็น* หากมีรายการใดตรงกับ `source_language` ระบบจะข้ามภาษานั้นโดยอัตโนมัติ ดังนั้นการใส่ภาษาต้นทางไว้ในรายการนี้จึงไม่มีผลเสียใดๆ

ตัวอย่างเช่น หากต้องการแปล README ภาษาจีนให้เป็นภาษาอังกฤษและสเปน:

```json
{
  "source_language": "中文",
  "target_languages": ["English", "Español"]
}
```

หากไม่ได้ระบุ `target_languages` ตัว pipeline จะใช้รายการภาษามาตรฐาน 18 ภาษาที่มีมาให้ในตัว

ส่วน `api_key` จะมีความสำคัญก็ต่อเมื่อคุณเปิดใช้งาน "Require API Key" ในการตั้งค่า Developer server ของ LM Studio มิฉะนั้นให้ปล่อยไว้เป็น `"lm-studio"` ซึ่ง LM Studio จะเพิกเฉยต่อค่านี โปรดทราบว่าสิ่งนี้ใช้กับโหมด `pipeline` เท่านั้น เนื่องจากเป็นส่วนเดียวที่เรียกใช้ OpenAI-compatible endpoint ของ LM Studio โดยตรง ส่วนโหมด `serve` จะไม่คุยกับ API ของ LM Studio เพราะตัว LM Studio เองนั่นแหละที่เป็น MCP client ที่เรียกใช้งาน server นี้

## การใช้งาน (Usage)

```bash
./run.sh serve    /absolute/path/to/project   # stdio MCP server
./run.sh pipeline /absolute/path/to/project   # รัน main.py แบบ end-to-end
```

ทั้งสองโหมดต้องการ **พาธแบบสมบูรณ์ (absolute path)** ของโฟลเดอร์โปรเจกต์ที่มีไฟล์ `README.md` ที่ต้องการแปล สำหรับโหมด `pipeline` คุณสามารถละเว้นพาธไว้ได้ และระบบจะถามให้คุณกรอกภายหลัง หากคุณรันผ่านเทอร์มินัลแบบโต้ตอบ:

```bash
./run.sh pipeline
Enter absolute path to project folder: /absolute/path/to/project
```

(โหมด `serve` จะไม่มีการถามข้อมูล เนื่องจากเมื่อเริ่มทำงาน stdin/stdout จะกลายเป็นช่องทางสื่อสาร MCP JSON-RPC และ MCP host จะเรียกใช้งานแบบ non-interactively อยู่แล้ว)

### `serve` — ในฐานะเครื่องมือของ MCP host

นี่คือส่วนที่ MCP host (เช่น LM Studio) ควรชี้คำสั่ง `command` มาที่นี่ โดยระบุพาธสมบูรณ์ของโปรเจกต์เป้าหมายเป็นอาร์กิวเมนต์คงที่ ซึ่งจะเปิดให้ใช้งาน:

- **tool** `write_readme(language, content)` — เขียนไฟล์ `docs/<language>/README.md`
- **tool** `write_directory_readme(content)` — เขียนไฟล์ `README.md` ที่ root (หน้าแลนดิ้งเพจเลือกภาษา)
- **resource** `docs://readme` — ไฟล์ต้นฉบับ (`docs/<source_language>/README.md` หรือ fallback ไปที่ root `README.md`)
- **resource** `docs://readme/{language}` — คำแปลที่มีอยู่ (ถ้ามี)
- **resource** `docs://dir_readme` — ไฟล์ `README.md` ที่ root
- **prompts** `translate_readme`, `critique_translation`, `rewrite_translation` — ลูปสำหรับการแปลใหม่
- **prompts** `check_existing_readme`, `rewrite_from_existing_translation` — เส้นทางการอัปเดต: เปรียบเทียบคำแปลเดิมกับต้นฉบับปัจจุบัน และแก้ไขเฉพาะจุดที่จำเป็นด้วยการเปลี่ยนแปลงให้น้อยที่สุด
- **prompt** `create_docs_language_directory` — สร้างหน้าเลือกภาษาที่ root จากรายชื่อคำแปลที่มีอยู่

โมเดลของตัว host เองจะเป็นผู้ขับเคลื่อนการเรียกใช้ tool ส่วน server นี้ทำหน้าที่เพียงจัดการ I/O ของไฟล์และเตรียมเทมเพลต prompt เท่านั้น

#### การเพิ่มเข้าไปใน LM Studio

การตั้งค่า MCP ของ LM Studio อยู่ที่ `~/.lmstudio/mcp.json` ให้เพิ่มรายการดังนี้:

```json
{
  "mcpServers": {
    "comprehend-markdown": {
      "command": "/absolute/path/to/comprehend-markdown/run.sh",
      "args": [
        "serve",
        "/absolute/path/to/the/project/you/want/to/translate"
      ]
    }
  }
}
```

อาร์กิวเมนต์ตัวที่สองใน `args` จะถูกกำหนดตายตัวตอนเชื่อมต่อ เนื่องจาก config ของ LM Studio เป็น JSON แบบ static และไม่รองรับการโต้ตอบ ดังนั้นต้องระบุพาธของโปรเจกต์ที่คุณต้องการแปลจริงๆ ไม่ใช่พาธของ repo นี้ (เว้นแต่คุณต้องการแปล README ของ repo นี้เอง)

### `pipeline` — ทำงานแบบ standalone โดยไม่ต้องมี MCP host

รันลูป แปล $\rightarrow$ วิพากษ์ $\rightarrow$ แก้ไข แบบเต็มรูปแบบสำหรับ `target_languages` ที่กำหนดไว้ (ค่าเริ่มต้น 18 ภาษา: Deutsch, Español, Français, Italiano, Polski, Português, Русский, Tiếng Việt, ไทย, 中文, 日本語, 한국어, العربية, हिन्दी, বাংলা, Bahasa Indonesia, اردو, Naijá — ดู [การเลือกภาษาต้นทางและปลายทาง](#การเลือกภาษาต้นทางและปลายทาง) เพื่อเปลี่ยนรายการหรือภาษาต้นทาง) โดยเรียกใช้ OpenAI-compatible endpoint ของ LM Studio โดยตรงสำหรับขั้นตอนการเขียน/ตรวจทาน และสร้าง `server.py` ภายในผ่าน stdio เพื่อจัดการ I/O ของไฟล์ เหมาะสำหรับการแปลจำนวนมากโดยไม่ต้องสั่งงานผ่าน UI ของ LM Studio ระยะเวลาในการทำงานจะขึ้นอยู่กับขนาดของเอกสารและโมเดลที่ใช้ — ใช้เวลาไม่กี่นาทีสำหรับ README ขนาดเล็ก และอาจถึงหนึ่งชั่วโมงสำหรับไฟล์ขนาดใหญ่ที่ต้องแบ่งส่วน

ภาษาที่มีไฟล์ `docs/<language>/README.md` อยู่แล้วจะไม่ถูกแปลใหม่ทั้งหมด แต่ผู้ตรวจทาน (reviewer) จะเปรียบเทียบคำแปลเดิมกับต้นฉบับปัจจุบัน และหากมีการเปลี่ยนแปลง ผู้เขียน (writer) จะแก้ไขเฉพาะจุดที่จำเป็นเท่านั้น คำแปลที่เป็นปัจจุบันอยู่แล้วจะถูกข้ามไป ดังนั้นการรัน pipeline ซ้ำหลังจากแก้ไข README จะทำเฉพาะส่วนที่ล้าสมัยเท่านั้น

หลังจากทำงานครบทุกภาษา ระบบจะปิดท้ายด้วยการย้าย `README.md` ต้นฉบับจาก root ไปยัง `docs/<source_language>/` (ถ้าจำเป็น) และสร้าง `README.md` ที่ root ใหม่ให้เป็นหน้าเลือกภาษาสั้นๆ ที่ลิงก์ไปยังคำแปลที่ไม่ว่างเปล่าทุกภาษา

ต้องเปิดใช้งาน local server ของ LM Studio ไว้ (ดู `config.json`) พร้อมโหลด chat model ใดก็ได้ — ตัว pipeline จะขับเคลื่อนโมเดลด้วย plain text completions และเขียนไฟล์เอง ดังนั้นโมเดล **ไม่จำเป็น** ต้องรองรับ OpenAI-style tool calls (ดูหัวข้อ "การจัดการ README ขนาดใหญ่" ด้านล่าง) อย่างไรก็ตาม แนะนำให้ทดสอบกับหนึ่งภาษาก่อนจะรันเต็มรูปแบบทุกภาษา

### การจัดการ README ขนาดใหญ่ (Large-README handling)

การแปล README ขนาดใหญ่ (เช่น เอกสาร `A-Starry-Sky` / `a-restless-ocean` ที่มีขนาด ~35–60 KB) ในการตอบกลับครั้งเดียว เป็นความเสี่ยงหลักด้านความเสถียรของโมเดลในเครื่อง เนื่องจากอาจเกิดปัญหา output tokens หรือ context เต็มกลางคัน ทำให้ส่วนท้ายถูกตัดออกโดยไม่มีการแจ้งเตือน โหมด `pipeline` จึงถูกออกแบบมาเพื่อป้องกันปัญหานี้:

- **ออกแบบมาให้ไม่ใช้ Tool (Tool-free by design):** โมเดลจะสร้างเพียงข้อความธรรมดา (plain text) เท่านั้น — ทุกคำแปล, การแก้ไข และหน้าไดเรกทอรี จะส่งกลับมาเป็นคำตอบของโมเดล และตัว orchestrator จะบันทึกไฟล์เองผ่าน tool `write_readme` / `write_directory_readme` ของ server วิธีนี้จะไม่บังคับให้โมเดลต้องยัดเอกสารทั้งฉบับลงในอาร์กิวเมนต์ของ tool call ซึ่งในโมเดลสาย *reasoning* (เช่น Gemma 4) การทำเช่นนั้นจะทำให้ JSON ของอาร์กิวเมนต์ถูกตัด และ endpoint จะปฏิเสธด้วยข้อผิดพลาด `peg-gemma4 format` / malformed-output การส่งกลับเป็นข้อความจึงช่วยเลี่ยงปัญหานี้ได้อย่างสมบูรณ์
- **`max_tokens`** (ค่าเริ่มต้น `32768`) จะถูกส่งไปอย่างชัดเจนในการตอบกลับทุกครั้ง เพื่อให้เนื้อหาแต่ละส่วนหรือฉบับร่างเสร็จสมบูรณ์ แทนที่จะถูกตัดตามค่าเริ่มต้นที่น้อยกว่าของ LM Studio หากการตอบกลับยังคงหยุดเพราะ `length` ระบบจะแจ้งเตือนเพื่อให้คุณเพิ่มค่านี้
- **การแบ่งส่วน (Section chunking):** เมื่อต้นฉบับมีขนาดเกิน `chunk_threshold_chars` (ค่าเริ่มต้น `12000`) และตั้งค่า `chunk_translation` เป็น `true` ต้นฉบับจะถูกแบ่งตามหัวข้อ Markdown ระดับบนสุด (`## `) โดยระบบจะคำนึงถึง code fence (ดังนั้น `##` ภายในบล็อกโค้ดจะไม่ถูกนำมาแบ่งส่วน) แต่ละส่วนจะถูกแปลแยกกันในการตอบกลับหนึ่งครั้ง จากนั้น orchestrator จะรวบรวมชิ้นส่วนทั้งหมดและบันทึกไฟล์ ทำให้ไม่มีการเรียกโมเดลครั้งเดียวเพื่อสร้างเอกสารทั้งฉบับ
- **การตรวจทานรายส่วน (Per-section review):** เมื่อตั้งค่า `review_sections` เป็น `true` (ค่าเริ่มต้น) แต่ละส่วนจะผ่านลูป วิพากษ์ $\rightarrow$ แก้ไข แบบเดียวกับวิธีปกติ แต่จำกัดขอบเขตอยู่แค่ในส่วนนั้นๆ ผู้ตรวจทานจะเปรียบเทียบส่วนที่แปลแล้วกับต้นฉบับ และผู้เขียนจะแก้ไขจนกว่าผู้ตรวจทานจะเห็นชอบ (หรือจนกว่าจะถึง `MAX_ITERATIONS`) เนื่องจากแต่ละส่วนมีขนาดเล็ก การตรวจทานและแก้ไขจึงไม่เสี่ยงต่อการถูกตัดเนื้อหาเหมือนกับการตรวจทานเอกสารขนาดใหญ่ทั้งฉบับ

ในเส้นทางการแบ่งส่วน ระบบจะ **ไม่** รันการตรวจทานทั้งฉบับซ้ำอีกครั้งในตอนท้าย เพราะการส่ง README ขนาดใหญ่ทั้งหมดกลับมาในการตอบกลับครั้งเดียวจะทำให้เกิดปัญหาการตัดเนื้อหาแบบเดิม และการตรวจทานรายส่วนได้ครอบคลุมเนื้อหาทั้งหมดแล้ว คุณสามารถตั้งค่า `review_sections` เป็น `false` เพื่อแปลเอกสารขนาดใหญ่โดยไม่มีการตรวจทาน หรือตั้งค่า `chunk_translation` เป็น `false` เพื่อบังคับให้ทำงานแบบครั้งเดียว (ซึ่งจะยังคงรันลูปตรวจทานทั้งฉบับ) สำหรับไฟล์ที่ขนาดต่ำกว่าเกณฑ์ การทำงานจะยังคงเป็นแบบเดิมไม่เปลี่ยนแปลง
