# comprehend-markdown

*آپ اس طومار کو چھوتے ہیں اور منتر پڑھتے ہیں۔ چند منٹ سے ایک گھنٹے بعد (طومار کی وسعت کے مطابق)، آپ کسی بھی زبان میں مارک ڈاؤن (Markdown) سمجھنے لگتے ہیں۔*

یہ ایک MCP سرور ہے جو کسی پروجیکٹ کی `README.md` فائل کا دوسری زبانوں میں ترجمہ کرتا ہے، اور ساتھ ہی ایک اسٹینڈ-الون پائپ لائن بھی فراہم کرتا ہے جو مقامی LM Studio ماڈل کا استعمال کرتے ہوئے رائٹر/ریویوور (writer/reviewer) لوپ کے ذریعے کام کرتی ہے۔

ماخذ (source) اور ہدف (target) دونوں زبانیں قابلِ تبدیلی ہیں — انگریزی صرف ڈیفالٹ زبان ہے۔ کسی دوسری زبان (جیسے چینی یا انڈونیشیائی) سے ترجمہ کرنے کے لیے `source_language` سیٹ کریں، اور جن زبانوں میں ترجمہ کرنا ہے ان کے لیے `target_languages` کا انتخاب کریں (دیکھیں [زبانوں کا انتخاب](#choosing-source-and-target-languages))۔

کسی بھی ہدف پروجیکٹ کے لیے، یہ درج ذیل ڈھانچے کی توقع رکھتا ہے (اور ضرورت پڑنے پر اسے خود بناتا ہے):

```
<project-root>/docs/<source>/README.md  بنیادی ماخذ (ڈیفالٹ طور پر انگریزی)
<project-root>/docs/<lang>/README.md    ترجمہ شدہ ورژن، ہر زبان کے لیے ایک فائل
<project-root>/README.md                زبان منتخب کرنے والا لینڈنگ پیج (خودکار طریقے سے تیار کردہ)
```

ایسے پروجیکٹس جو ابھی تک منتقل نہیں ہوئے — یعنی جن کا ماخذ اب بھی روٹ (root) میں ہے — وہ بھی کام کریں گے: روٹ کی `README.md` کو بطور ماخذ استعمال کیا جاتا ہے، اور `pipeline` کے عمل کے اختتام پر اسے `docs/<source>/` (مثلاً `docs/English/`) میں منتقل کر دیا جاتا ہے اور اس کی جگہ ایک مختصر لینڈنگ پیج بنا دیا جاتا ہے جو تمام دستیاب تراجم کے لنکس فراہم کرتا ہے۔

## سیٹ اپ (Setup)

تمام مراحل (venv کی تخلیق، ڈیپینڈینسیز کی انسٹالیشن/سنک) `run.sh` کے ذریعے سنبھالے جاتے ہیں — انسٹالیشن کا کوئی علیحدہ مرحلہ نہیں ہے۔ یہ `requirements.txt` سے پیکجز پڑھتا ہے اور ہر بار چلنے پر انہیں دوبارہ انسٹال کرتا ہے، لہذا نئی ڈیپینڈینسیز حاصل کرنے کے لیے بس اسے دوبارہ چلانا کافی ہے۔

اگر آپ ڈیفالٹ کے بجائے کسی مقامی LM Studio ماڈل کی نشاندہی کرنا چاہتے ہیں، تو مثال کے طور پر دی گئی کنفیگریشن فائل کو کاپی کریں اور اسے ایڈٹ کریں:

```bash
cp config.local.json.example config.local.json
```

`config.local.json` فائل gitignored ہے اور یہ `config.json` کی ویلیوز کو اوور رائیڈ (override) کرتی ہے، لہذا آپ کمیٹ شدہ ڈیفالٹس کو چھیڑے بغیر مقامی طور پر `lm_studio_url` / `model_name` / `api_key` میں تبدیلیاں کر سکتے ہیں۔

### زبانوں کا انتخاب

دو کنفیگ کیز ترجمے کی سمت کو کنٹرول کرتی ہیں، اور سرور اور پائپ لائن دونوں انہیں (`config.py` سے) پڑھتے ہیں تاکہ ان میں کوئی تضاد نہ ہو:

- **`source_language`** — وہ زبان جس میں آپ کی بنیادی `README.md` لکھی گئی ہے، اور اس کے فولڈر `docs/<source_language>/` کا نام۔ ڈیفالٹ طور پر یہ `"English"` ہے۔ اسے `"中文"`, `"Bahasa Indonesia"`, یا کسی بھی دوسری زبان پر سیٹ کریں تاکہ ترجمہ اس زبان سے شروع ہو۔
- **`target_languages`** — ان زبانوں کی فہرست جن میں پائپ لائن ترجمہ کرتی ہے۔ اگر کوئی اندراج `source_language` کے برابر ہو تو اسے خود بخود چھوڑ دیا جاتا ہے، لہذا فہرست میں ماخذ زبان کا ہونا نقصان دہ نہیں ہے۔

مثال کے طور پر، چینی README کا انگریزی اور ہسپانوی میں ترجمہ کرنے کے لیے:

```json
{
  "source_language": "中文",
  "target_languages": ["English", "Español"]
}
```

اگر `target_languages` فراہم نہیں کی گئی، تو پائپ لائن اپنی بلٹ-ان 18 زبانوں کی فہرست استعمال کرے گی۔

`api_key` صرف تب اہمیت رکھتی ہے جب آپ نے LM Studio کی Developer سرور سیٹنگز میں "Require API Key" کو آن کیا ہو — ورنہ اسے `"lm-studio"` ہی رہنے دیں، جسے LM Studio نظر انداز کر دیتا ہے۔ نوٹ کریں کہ یہ صرف `pipeline` موڈ پر لاگو ہوتا ہے کیونکہ یہی وہ حصہ ہے جو براہ راست LM Studio کے OpenAI-مطابق اینڈ پوائنٹ (endpoint) کو کال کرتا ہے؛ `serve` موڈ کبھی بھی LM Studio کی API سے بات نہیں کرتا کیونکہ LM Studio خود MCP کلائنٹ ہے جو اسے کال کر رہا ہوتا ہے۔

## استعمال (Usage)

```bash
./run.sh serve    /absolute/path/to/project   # stdio MCP server
./run.sh pipeline /absolute/path/to/project   # runs main.py end-to-end
```

دونوں موڈز کے لیے پروجیکٹ فولڈر کا **مکمل (absolute)** راستہ ضروری ہے جس میں ترجمہ کی جانے والی `README.md` موجود ہو۔ `pipeline` موڈ میں، اگر آپ اسے ٹرمینل پر انٹرایکٹو طریقے سے چلا رہے ہیں تو آپ راستہ چھوڑ سکتے ہیں اور سسٹم آپ سے اس کا مطالبہ کرے گا:

```bash
./run.sh pipeline
Enter absolute path to project folder: /absolute/path/to/project
```

(`serve` موڈ کبھی بھی پراپٹ نہیں کرتا — ایک بار شروع ہونے کے بعد، stdin/stdout خود MCP JSON-RPC چینل بن جاتے ہیں، اور MCP ہوسٹس اسے غیر انٹرایکٹو طریقے سے لانچ کرتے ہیں۔)

### `serve` — بطور MCP host tool

یہ وہ کمانڈ ہے جس کی طرف ایک MCP ہوسٹ (مثلاً LM Studio) کو اپنے سرور `command` میں اشارہ کرنا چاہیے، جس میں ٹارگٹ پروجیکٹ کا مکمل راستہ ایک فکسڈ آرگیومنٹ کے طور پر ہو۔ یہ درج ذیل چیزیں فراہم کرتا ہے:

- **tool** `write_readme(language, content)` — (README لکھنے کے لیے) `docs/<language>/README.md` لکھتا ہے۔
- **tool** `write_directory_readme(content)` — (لینڈنگ پیج لکھنے کے لیے) روٹ کی `README.md` لکھتا ہے۔
- **resource** `docs://readme` — (بنیادی ماخذ) `docs/<source_language>/README.md` تک رسائی دیتا ہے، یا روٹ کی `README.md` پر واپس جاتا ہے۔
- **resource** `docs://readme/{language}` — (موجودہ ترجمہ) اگر کوئی ترجمہ موجود ہو تو اسے دکھاتا ہے۔
- **resource** `docs://dir_readme` — (روٹ فائل) روٹ کی `README.md` تک رسائی دیتا ہے۔
- **prompts** `translate_readme`, `critique_translation`, `rewrite_translation` — (ترجمہ لوپ) نئے ترجمے کا عمل۔
- **prompts** `check_existing_readme`, `rewrite_from_existing_translation` — (اپ ڈیٹ کا راستہ) موجودہ ترجمے کا موازنہ حالیہ ماخذ سے کرنا اور کم سے کم تبدیلیوں کے ساتھ اسے درست کرنا۔
- **prompt** `create_docs_language_directory` — (لینڈنگ پیج کی تخلیق) دستیاب تراجم کی فہرست سے روٹ لینگویج پکر پیج بنانا۔

ہوسٹ کا اپنا ماڈل ٹول کالز کو کنٹرول کرتا ہے؛ یہ سرور صرف فائل I/O اور پرامپٹ ٹیمپلیٹس فراہم کرتا ہے۔

#### اسے LM Studio میں شامل کرنا

LM Studio کی MCP کنفیگریشن `~/.lmstudio/mcp.json` پر موجود ہوتی ہے۔ اس میں ایک اندراج اس طرح شامل کریں:

```json
{
  "mcpServers": {
    "comprehend-markdown": {
      "command": "/absolute/path/to/comprehend-markdown/run.sh",
      "args": [
        "serve",
        "/absolute/path/to/the/project/you/want/to/translate"
      ]
    }
  }
}
```

دوسرا `args` اندراج کنکشن کے وقت فکسڈ ہوتا ہے — LM Studio کی کنفیگریشن ایک اسٹیٹک JSON ہے جس میں انٹرایکٹو پراپٹ کی سپورٹ نہیں ہے، لہذا یہاں اس پروجیکٹ کا اصل راستہ ہونا چاہیے جس کا آپ ترجمہ کرنا چاہتے ہیں، نہ کہ اس ریپوزٹری کا اپنا راستہ (بشرطیکہ آپ اسی کا ترجمہ کرنا چاہیں)۔

### `pipeline` — اسٹینڈ-الون، بغیر MCP host کے

یہ کنفیگر شدہ `target_languages` کے لیے مکمل translate $\rightarrow$ critique $\rightarrow$ revise لوپ خود چلاتا ہے (ڈیفالٹ 18 زبانوں کی فہرست: Deutsch, Español, Français, Italiano, Polski, Português, Русский, Tiếng Việt, ไทย, 中文, 日本語, 한국어, العربية, हिन्दी, বাংলা, Bahasa Indonesia, اردو, Naijá — فہرست یا ماخذ زبان تبدیل کرنے کے لیے [زبانوں کا انتخاب](#choosing-source-and-target-languages) دیکھیں)۔ یہ رائٹر/ریویوور کے مراحل کے لیے براہ راست LM Studio کے OpenAI-مطابق اینڈ پوائنٹ کو کال کرتا ہے اور فائل I/O کے لیے stdio پر `server.py` کی اپنی اندرونی کاپی استعمال کرتا ہے۔ یہ LM Studio UI کے بغیر بیچ-ترجمہ (batch-translating) کرنے کے لیے مفید ہے۔ مقامی ماڈل پر رن ٹائم دستاویز کے سائز کے مطابق ہوتا ہے — چھوٹی README کے لیے چند منٹ، اور بڑی دستاویزات کے لیے ایک گھنٹہ یا اس سے زیادہ۔

وہ زبانیں جن کی `docs/<language>/README.md` پہلے سے موجود ہے، انہیں شروع سے دوبارہ ترجمہ نہیں کیا جاتا: ریویوور موجودہ ترجمے کا موازنہ حالیہ ماخذ سے کرتا ہے اور صرف تب جب کچھ تبدیل ہوا ہو، رائٹر اسے کم سے کم تبدیلیوں کے ساتھ درست کرتا ہے۔ اپ-ٹو-ڈیٹ تراجم کو مکمل طور پر چھوڑ دیا جاتا ہے، لہذا README میں ترمیم کے بعد پائپ لائن دوبارہ چلانے سے صرف وہی حصے دوبارہ ترجمہ ہوتے ہیں جو پرانے ہو چکے ہوں۔

ہر زبان کا کام مکمل ہونے کے بعد، یہ عمل (اگر ضرورت ہو) روٹ لیول کی `README.md` کو `docs/<source_language>/` میں منتقل کرتا ہے اور روٹ کی `README.md` کو ایک مختصر لینگویج پکر پیج کے طور پر دوبارہ تیار کرتا ہے جو تمام غیر خالی تراجم کے لنکس فراہم کرتا ہے۔

اس کے لیے LM Studio کا مقامی سرور چلنا ضروری ہے (دیکھیں `config.json`) جس میں کوئی بھی چیٹ ماڈل لوڈ ہو — پائپ لائن سادہ ٹیکسٹ کمپلیشنز (plain text completions) کے ذریعے ماڈل کو چلاتا ہے اور فائلیں خود لکھتا ہے، لہذا ماڈل کے لیے OpenAI اسٹائل ٹول کالز کی سپورٹ ہونا **ضروری نہیں** (نیچے "Large-README handling" دیکھیں)۔ پھر بھی مشورہ ہے کہ پوری فہرست پر بھروسہ کرنے سے پہلے ایک زبان کا تجربہ کر لیں۔

### Large-README handling (بڑی README فائلز کا انتظام)

ایک بڑی README (مثلاً `A-Starry-Sky` / `a-restless-ocean` کی دستاویزات جو تقریباً 35–60 KB ہیں) کا ایک ہی بار میں ترجمہ کرنا مقامی ماڈل پر سب سے بڑا خطرہ ہے: یہ درمیان میں آؤٹ پٹ ٹوکنز یا کانٹیکسٹ ختم ہونے کی وجہ سے خاموشی سے کٹ سکتا ہے۔ `pipeline` موڈ کو اسی مسئلے سے بچنے کے لیے بنایا گیا ہے:

- **ڈیزائن کے لحاظ سے ٹول-فری (Tool-free by design):** ماڈل صرف سادہ ٹیکسٹ تیار کرتا ہے — ہر ترجمہ، ری رائٹ اور ڈائریکٹری پیج اس کے جواب کے طور پر واپس آتا ہے، اور آرکسٹریٹر اسے سرور کے `write_readme` / `write_directory_readme` ٹولز کے ذریعے خود محفوظ کرتا ہے۔ ماڈل سے یہ مطالبہ نہیں کیا جاتا کہ وہ پوری دستاویز کو ایک ٹول-کال آرگیومنٹ میں بھرے۔ مقامی *reasoning* ماڈلز (مثلاً Gemma 4) پر، ایسا کرنے سے آرگیومنٹ JSON کٹ جاتا ہے اور اینڈ پوائنٹ اسے `peg-gemma4 format` / malformed-output ایرر کے ساتھ مسترد کر دیتا ہے؛ سادہ ٹیکسٹ واپس لینا اس مسئلے کو مکمل طور پر ختم کر دیتا ہے۔
- **`max_tokens`** (ڈیفالٹ `32768`) ہر کمپلیشن پر واضح طور پر بھیجا جاتا ہے تاکہ ایک پورا سیکشن یا ڈرافٹ مکمل ہو سکے، بجائے اس کے کہ وہ LM Studio کے چھوٹے ڈیفالٹ پر کٹ جائے۔ اگر کوئی کمپلیشن پھر بھی `length` کی وجہ سے رکتا ہے تو پائپ لائن وارننگ دیتا ہے تاکہ آپ اسے بڑھا سکیں۔
- **سیکشن چنکنگ (Section chunking):** جب ماخذ `chunk_threshold_chars` (ڈیفالٹ `12000`) سے بڑھ جائے اور `chunk_translation` کی ویلیو `true` ہو، تو ماخذ کو ٹاپ لیول (`## `) مارک ڈاؤن ہیڈنگز پر تقسیم کیا جاتا ہے (یہ فینس-اویر ہے، لہذا کوڈ بلاک کے اندر موجود `##` کو نہیں چھیڑا جاتا ہے)، ہر سیکشن کا ترجمہ اپنی الگ کمپلیشن میں ہوتا ہے، اور آرکسٹریٹر ان حصوں کو جوڑ کر محفوظ کرتا ہے۔ کسی بھی سنگل ماڈل کال کو پوری دستاویز جاری کرنے کی ضرورت نہیں پڑتی۔
- **فی-سیکشن ریویو (Per-section review):** جب `review_sections` کی ویلیو `true` ہو (جو کہ ڈیفالٹ ہے)، تو ہر سیکشن اسی critique $\rightarrow$ revise لوپ سے گزرتا ہے جو سنگل-شاٹ راستے میں استعمال ہوتا ہے، لیکن صرف اس ایک سیکشن تک محدود رہتا ہے: ریویوور ترجمہ شدہ سیکشن کا موازنہ اس کے ماخذ سے کرتا ہے اور رائٹر تب تک اسے درست کرتا ہے جب تک ریویوور متفق نہ ہو جائے (یا `MAX_ITERATIONS` مکمل نہ ہو جائیں)۔ چونکہ ایک سیکشن چھوٹا ہوتا ہے، اس لیے ریویو اور ری رائٹ اس کٹنگ کی حد (truncation ceiling) سے بہت دور رہتے ہیں جس نے بڑی README کے پورے دستاویز ریویو کو غیر محفوظ بنا دیا تھا۔

چنکڈ راستہ جان بوجھ کر بعد میں دوسرا مکمل دستاویز ریویو **نہیں** کرتا — کیونکہ ایک ہی کمپلیشن میں پوری بڑی README کو دوبارہ جاری کرنا وہی کٹنگ کا مسئلہ پیدا کرے گا، اور فی-سیکشن پاس پہلے ہی اسے کور کر چکا ہوتا ہے۔ اگر آپ بغیر کسی ریویو کے بڑی دستاویزات کا ترجمہ کرنا چاہتے ہیں تو `review_sections` کو `false` پر سیٹ کریں، یا اصل سنگل-شاٹ رویے کو زبردستی نافذ کرنے کے لیے `chunk_translation` کو `false` کر دیں (جو اب بھی مکمل دستاویز ریویو لوپ چلاتا ہے)۔ تھریشولڈ سے نیچے والے راستے میں کوئی تبدیلی نہیں کی گئی ہے۔
