# comprehend-markdown

*آپ اس طومار کو چھوتے ہیں اور جادوئی الفاظ ادا کرتے ہیں۔ چند منٹ سے ایک گھنٹے بعد (دستاویز کی لمبائی کے مطابق)، آپ کسی بھی زبان میں مارک ڈاؤن (Markdown) کو سمجھنے لگتے ہیں۔*

یہ ایک MCP سرور ہے جو کسی پروجیکٹ کی `README.md` فائل کا دوسری زبانوں میں ترجمہ کرتا ہے، اور ساتھ ہی ایک اسٹینڈالون پائپ لائن بھی فراہم کرتا ہے جو مقامی LM Studio ماڈل کا استعمال کرتے ہوئے رائٹر/ریویوئر (writer/reviewer) لوپ کے ذریعے کام کرتی ہے۔

ذریعہ (source) اور ہدف (target) دونوں زبانیں ترتیب دی جا سکتی ہیں — انگریزی صرف ڈیفالٹ زبان ہے۔ اگر آپ چینی، انڈونیشیائی یا کسی اور زبان سے ترجمہ کرنا چاہتے ہیں تو `source_language` کو تبدیل کریں، اور جن زبانوں میں ترجمہ کرنا ہے انہیں `target_languages` میں منتخب کریں (تفصیل کے لیے [زبانوں کا انتخاب](#choosing-source-and-target-languages) دیکھیں)۔

کسی بھی ہدف پروجیکٹ کے لیے، یہ درج ذیل ڈھانچے کی توقع رکھتا ہے (اور ضرورت پڑنے پر اسے خود بنا لیتا ہے):

```
<project-root>/docs/<source>/README.md  اصل بنیادی ذریعہ (ڈیفالٹ طور پر انگریزی)
<project-root>/docs/<lang>/README.md    ترجمہ شدہ ورژن، ہر زبان کے لیے ایک فائل
<project-root>/README.md                زبان منتخب کرنے والا لینڈنگ پیج (خودکار طریقے سے تیار کردہ)
```

ایسے پروجیکٹس جو ابھی تک منتقل نہیں ہوئے — یعنی جن کا ذریعہ اب بھی روٹ (root) میں ہے — وہ بھی کام کریں گے: ایسی صورت میں روٹ کی `README.md` کو بطور ذریعہ استعمال کیا جاتا ہے، اور `pipeline` کے عمل کے اختتام پر اسے `docs/<source>/` (مثلاً `docs/English/`) میں منتقل کر دیا جاتا ہے اور اس کی جگہ ایک مختصر لینڈنگ پیج بنا دیا جاتا ہے جو تمام دستیاب تراجم کے لنکس فراہم کرتا ہے۔

## سیٹ اپ (Setup)

تمام معاملات (venv کی تخلیق، ڈیپینڈینسیز کا انسٹالیشن/سنک) `run.sh` کے ذریعے سنبھالے جاتے ہیں — انسٹالیشن کے لیے کسی علیحدہ مرحلے کی ضرورت نہیں ہے۔ یہ `requirements.txt` سے پیکجز پڑھتا ہے اور ہر بار چلنے پر انہیں دوبارہ انسٹال کرتا ہے، لہذا نئی ڈیپینڈینسیز حاصل کرنے کا مطلب صرف اسے دوبارہ چلانا ہے۔

اگر آپ ڈیفالٹ کے بجائے کسی مقامی LM Studio ماڈل کو استعمال کرنا چاہتے ہیں، تو `config.json` میں موجود ویلیوز (values) میں تبدیلی کریں۔

### زبانوں کا انتخاب (Choosing source and target languages)

`config.json` میں دو کنفیگ کیز (keys) ترجمے کی سمت کو کنٹرول کرتی ہیں:

- **`source_language`** — وہ زبان جس میں آپ کی اصل `README.md` لکھی گئی ہے، اور اس کے فولڈر `docs/<source_language>/` کا نام۔ ڈیفالٹ طور پر یہ `English` ہے۔ اسے `中文` ، `Indonesia` یا کسی بھی دوسری زبان پر سیٹ کریں تاکہ ترجمہ اس زبان سے شروع ہو۔
- **`target_languages`** — ان زبانوں کی فہرست جن میں پائپ لائن ترجمہ کرے گی۔ اگر کوئی اندراج `source_language` کے برابر ہوا تو اسے خود بخود چھوڑ دیا جائے گا، لہذا لسٹ میں اصل زبان کا ہونا کوئی مسئلہ نہیں ہے۔

مثال کے طور پر، چینی README کا انگریزی اور ہسپانوی میں ترجمہ کرنے کے لیے:

```json
{
  "source_language": "中文",
  "target_languages": ["English", "Español"]
}
```

اگر `target_languages` فراہم نہیں کی گئی، تو پائپ لائن اپنی بلٹ-ان 18 زبانوں کی فہرست استعمال کرے گی۔

`api_key` صرف اس صورت میں ضروری ہے اگر آپ نے LM Studio کی Developer server سیٹنگز میں "Require API Key" کو آن کیا ہو — ورنہ اسے `"lm-studio"` ہی رہنے دیں، جسے LM Studio نظر انداز کر دیتا ہے۔ نوٹ کریں کہ یہ صرف `pipeline` موڈ پر لاگو ہوتا ہے، کیونکہ یہی وہ حصہ ہے جو براہ راست LM Studio کے OpenAI-مطابق اینڈ پوائنٹ (endpoint) کو کال کرتا ہے؛ `serve` موڈ کبھی بھی LM Studio کی API سے بات نہیں کرتا کیونکہ LM Studio خود MCP کلائنٹ ہے جو اسے کال کر رہا ہوتا ہے۔

`max_tokens` یہاں سیٹ کیا جا سکتا ہے، لیکن میں یقینی طور پر نہیں جانتا کہ LM Studio اس کی پابندی کرتا ہے یا نہیں۔ اس اسکرپٹ کو چلانے سے پہلے ماڈل کے اندر اسے کم از کم 24576 کی ویلیو پر سیٹ کرنا یقینی بنائیں۔

## استعمال (Usage)

```bash
./run.sh serve    /absolute/path/to/project   # stdio MCP server
./run.sh pipeline /absolute/path/to/project   # runs main.py end-to-end
```

دونوں موڈز کے لیے پروجیکٹ فولڈر کا **مکمل راستہ (absolute path)** ضروری ہے جس میں ترجمہ کی جانے والی `README.md` موجود ہو۔ `pipeline` موڈ میں، اگر آپ اسے ٹرمینل پر انٹرایکٹو طریقے سے چلا رہے ہیں تو آپ راستے کو چھوڑ سکتے ہیں اور سسٹم آپ سے اسے مانگے گا:

```bash
./run.sh pipeline
Enter absolute path to project folder: /absolute/path/to/project
```

(`serve` موڈ کبھی بھی راستہ نہیں مانگتا — ایک بار شروع ہونے کے بعد، stdin/stdout خود MCP JSON-RPC چینل بن جاتے ہیں، اور MCP ہوسٹس اسے غیر انٹرایکٹو طریقے سے لانچ کرتے ہیں۔)

### `serve` — بطور MCP host tool

یہ وہ کمانڈ ہے جس کی طرف ایک MCP ہوسٹ (مثلاً LM Studio) کو اپنے سرور `command` میں اشارہ کرنا چاہیے، جس میں ٹارگٹ پروجیکٹ کا مکمل راستہ ایک فکسڈ آرگیومنٹ کے طور پر دیا گیا ہو۔ یہ درج ذیل چیزیں فراہم کرتا ہے:

- **tool** `write_readme(language, content)` — (README لکھنے کے لیے) `docs/<language>/README.md` لکھتا ہے۔
- **tool** `write_directory_readme(content)` — (مرکزی صفحہ لکھنے کے لیے) روٹ کی `README.md` لکھتا ہے۔
- **resource** `docs://readme` — اصل ذریعہ (`docs/<source_language>/README.md` یا روٹ کی `README.md`)۔
- **resource** `docs://readme/{language}` — موجودہ ترجمہ، اگر کوئی ہو۔
- **resource** `docs://dir_readme` — روٹ کی `README.md`۔
- **prompts** `translate_readme`, `critique_translation`, `rewrite_translation` — نئے ترجمے کا لوپ۔
- **prompts** `check_existing_readme`, `rewrite_from_existing_translation` — اپ ڈیٹ کا راستہ: موجودہ ترجمے کا موازنہ اصل ذریعہ سے کرنا اور کم سے کم تبدیلیوں کے ساتھ اسے درست کرنا۔
- **prompt** `create_docs_language_directory` — دستیاب تراجم کی فہرست سے مرکزی زبان منتخب کرنے والا صفحہ بنانا۔

ہوسٹ کا اپنا ماڈل ٹول کالز کو کنٹرول کرتا ہے؛ یہ سرور صرف فائل I/O اور پرامپٹ ٹیمپلیٹس فراہم کرتا ہے۔

#### اسے LM Studio میں شامل کرنا

LM Studio کی MCP کنفیگ `~/.lmstudio/mcp.json` پر موجود ہوتی ہے۔ اس میں اس طرح کا اندراج شامل کریں:

```json
{
  "mcpServers": {
    "comprehend-markdown": {
      "command": "/absolute/path/to/comprehend-markdown/run.sh",
      "args": [
        "serve",
        "/absolute/path/to/the/project/you/want/to/translate"
      ]
    }
  }
}
```

دوسرا `args` اندراج کنکشن کے وقت طے شدہ ہوتا ہے — LM Studio کی کنفیگ ایک اسٹیٹک JSON فائل ہے جس میں انٹرایکٹو پرامپٹ کی سپورٹ نہیں ہے، اس لیے یہاں وہ اصل راستہ ہونا چاہیے جس کا آپ ترجمہ کرنا چاہتے ہیں، نہ کہ اس ریپوزٹری کا اپنا راستہ (بشرطیکہ آپ اسی کا ترجمہ کرنا چاہیں)۔
